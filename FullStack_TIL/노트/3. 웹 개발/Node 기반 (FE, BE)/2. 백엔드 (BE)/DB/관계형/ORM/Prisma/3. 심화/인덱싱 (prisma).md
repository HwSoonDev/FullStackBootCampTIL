**데이터베이스에서 인덱스 설정은 특정 컬럼을 빠르게 검색할 수 있도록 별도의 자료구조를 만드는 작업입니다.** 책의 목차나 도서관 색인처럼, 원하는 데이터를 효율적으로 찾게 해주는 기능이죠. 하지만 조회 속도를 높이는 대신, 삽입·수정·삭제 시에는 인덱스를 갱신해야 하므로 성능에 영향을 줄 수 있습니다.

📘 인덱스의 기본 개념

- **정의**: 테이블의 특정 컬럼 값과 해당 레코드의 물리적 위치를 매핑해둔 데이터베이스 객체.
- **비유**: 도서관에서 책을 찾을 때 책 번호(색인)를 활용하는 것과 동일.
- **구조**: 대부분의 DBMS는 **B-Tree 또는 B+Tree** 구조를 사용해 정렬된 상태를 유지.

⚡ 인덱스의 주요 역할

- **검색 속도 향상**: 조건 검색 시 전체 테이블을 스캔하지 않고 빠르게 탐색.
- **중복 방지**: Primary Key 인덱스를 통해 고유 값 보장.
- **정렬 최적화**: ORDER BY 같은 정렬 작업을 효율적으로 처리.

🔑 인덱스가 기본 설정되는 경우

- **Primary Key**: 각 행을 고유하게 식별하기 위해 자동으로 인덱스 생성.
- **Foreign Key**: 다른 테이블과의 관계를 빠르게 연결하기 위해 인덱스가 자주 사용됨.

📌 인덱스 사용 시 고려사항

- **효과적인 경우**: WHERE 조건에 자주 사용되는 컬럼, JOIN에 활용되는 컬럼.
- **피해야 할 경우**: 데이터가 자주 변경되는 컬럼, 소규모 테이블(전체 스캔이 더 빠름).
- **균형 필요**: 읽기 성능은 좋아지지만 쓰기 성능은 떨어질 수 있으므로, 조회 중심 시스템에 적합.


### Prisma에서의 인덱싱

**Prisma 스키마에서 인덱스 설정은 `@@index` 구문을 사용해 특정 필드(컬럼)에 대해 데이터베이스 인덱스를 정의하는 것입니다.** 이를 통해 검색·필터링 성능을 높일 수 있으며, 단일 컬럼뿐 아니라 여러 컬럼을 묶어 복합 인덱스를 만들 수도 있습니다.

📘 Prisma 스키마에서 인덱스 설정 방법

1. 기본 구문

```prsima.schema
model User {
  id       Int    @id @default(autoincrement())
  email    String @unique
  name     String
  password String

  @@index([name, email], name: "name_email_index")
}
```

- **`@@index([필드])`**: 지정한 필드에 인덱스를 생성.
- **`name: "인덱스명"`**: 인덱스 이름을 명시적으로 지정 가능.
- **복합 인덱스**: 여러 필드를 배열로 지정하면 다중 컬럼 인덱스 생성.

2. 인덱스 종류

- **`@unique`**: 단일 필드에 유니크 인덱스 생성. 중복 방지.
- **`@@unique([필드])`**: 여러 필드 조합에 대해 유니크 인덱스 생성.
- **`@@index([필드])`**: 일반 인덱스. 검색 성능 향상 목적.

2. 사용 예시

```
model Note {
  id        Int      @id @default(autoincrement())
  creator   String
  title     String
  content   String
  createdAt DateTime @default(now())
  userId    Int

  @@index([creator], name: "creator_index")
}
```

- `creator` 컬럼에 인덱스를 걸어 검색 속도를 향상.
- `@@index`는 조회 성능 최적화에 주로 사용.

⚡ 인덱스 설정 시 고려사항

- **조회 패턴 분석**: WHERE 조건, JOIN, ORDER BY에 자주 쓰이는 컬럼에 인덱스 설정.
- **쓰기 성능 영향**: 인덱스가 많으면 INSERT/UPDATE/DELETE 시 성능 저하.
- **테이블 크기**: 작은 테이블은 인덱스 효과가 미미할 수 있음.
- **복합 인덱스 활용**: 여러 조건을 동시에 검색할 때 유용.

✅ 요약

- Prisma 스키마에서 인덱스는 **`@@index`** 또는 **`@unique`** 구문으로 설정.
- **단일 컬럼**뿐 아니라 **복합 컬럼 인덱스**도 가능.
- 인덱스는 **검색 성능을 높이지만 쓰기 성능을 떨어뜨릴 수 있으므로 전략적으로 설정**해야 함.

#### 주요 효과

1. **검색 속도 향상**

- 인덱스가 걸린 컬럼을 기준으로 `WHERE`, `ORDER BY`, `JOIN` 같은 쿼리를 실행할 때 전체 테이블을 스캔하지 않고 인덱스를 통해 빠르게 탐색합니다.
- 예: `@@index([email])`를 설정하면 이메일로 사용자 검색 시 속도가 크게 개선됩니다.

2. **정렬 및 필터링 최적화**

- `ORDER BY createdAt` 같은 정렬 쿼리에서 인덱스가 있으면 DB가 이미 정렬된 구조를 활용해 결과를 빠르게 반환합니다.
- 대량 데이터에서 특히 효과적입니다.

3. **중복 방지 및 무결성 보장**

- `@unique` 또는 `@@unique([필드])` 인덱스를 설정하면 해당 컬럼(또는 컬럼 조합)에 중복 값이 들어갈 수 없습니다.
- 예: `email`에 `@unique`를 설정하면 같은 이메일을 가진 사용자가 두 번 저장되지 않습니다.

4. **복합 조건 최적화**

- 여러 컬럼을 묶은 복합 인덱스(`@@index([firstName, lastName])`)는 두 조건을 동시에 검색할 때 성능을 높입니다.
- 단일 인덱스보다 특정 검색 패턴에 더 최적화됩니다.

⚠️ 주의할 점

- **쓰기 성능 저하**: 인덱스는 데이터를 삽입·수정·삭제할 때마다 갱신해야 하므로, 인덱스가 많으면 쓰기 성능이 떨어질 수 있습니다.
- **저장 공간 증가**: 인덱스는 별도의 자료구조를 유지하므로 추가적인 디스크 공간을 사용합니다.
- **잘못된 인덱스 남발**: 조회 패턴과 맞지 않는 인덱스는 성능을 오히려 저하시킬 수 있습니다.
