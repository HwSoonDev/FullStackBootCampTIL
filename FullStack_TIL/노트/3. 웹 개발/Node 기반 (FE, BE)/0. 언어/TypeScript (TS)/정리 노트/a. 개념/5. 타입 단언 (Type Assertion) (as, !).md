### 타입 단언 (Type Assertion)

" as "

- 개발자가 TypeScript보다 더 정확한 타입 정보를 알고 있을 때 사용
- "나는 이 값의 타입을 확실히 알고 있어!"라고 컴파일러에게 알려주는 것
- 런타임에는 영향을 주지 않음 (컴파일 타임에만 사용)
- 주의: 잘못 사용하면 런타임 오류 발생 가능! 꼭 필요한 경우에만 사용할 것!

잘못된 타입 단언을 사용해서, **`컴파일러`**를 속이면 안되겠죠?

```tsx
// DOM 요소 조작 시 자주 사용
const inputElement = document.getElementById('username') as HTMLInputElement;
inputElement.value = 'hello'; // HTMLInputElement로 단언했으므로 value 접근 가능

// API 응답 처리 시
const apiResponse = {
  id: 1,
  name: "김개발",
  email: "kim@example.com"
};

interface User {
  id: number;
  name: string;
  email: string;
}

const user = apiResponse as User; // API 응답을 User 타입으로 단언
```

### _Non-null Assertion이란?_

" ! " 기호
- 값이 null이나 undefined가 아니라고 확신할 때 사용
- 변수 뒤에 ! 를 붙여서 사용
- 컴파일러의 null/undefined 체크를 무시
- 잘못 사용하면 런타임 오류 발생!

tip)
" ? " 기호 (옵셔널, 옵셔널 체이닝) 을 사용하는 슴관(버릇)이 좋은 것은 아니다.

```tsx
// 기본 사용법
let userName: string | null = "김개발";
console.log(userName!.toUpperCase()); // ! 로 null이 아님을 단언

// DOM 요소에서 자주 사용 (하지만 주의 필요!)
const button = document.getElementById('submit-btn')!; // element가 확실히 존재한다고 단언
button.addEventListener('click', handleClick);
```

```
1. 타입 단언은 최후의 수단으로 사용_
2. 가능하면 타입 가드나 조건문 사용 (런타임 검증)_
3. Non-null assertion(!)은 정말 확신할 때만 사용_
4. any를 사용하는 것보다는 타입 단언이 나음(타입 안정성을 일부 유지할 수 있으니까)
```💡
