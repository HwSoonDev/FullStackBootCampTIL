HTTP 기반 API 요청을 주고 받을 때, 요청 종류, 리퀘트와 리스폰스의 양식을 맞춘 것을 말한다.
- = URL로 자원을 식별하고, HTTP 메서드로 해당 자원에 대한 행위를 표현하는 구조

본래는 백엔드에서 JSON 기반 객체에 엔드포인트와 양식을 정의하여 프론트에 전달해줌으로서 해당 변수, 함수를 통해 백엔드 측 API요청 주소 변경 등에 영향을 받지 않고, 요청을 보낼 수 있도록 하는 말 그대로 프론트 입장에서 REST(편안한)한 방식을 지향한 것이 RESTful API의 정의였지만, 현재는 단순하게 양식만 맞춰서 보내는 식으로 정착했다.

요청 종류 (HTTP 메서드)

- GET
- POST
- PATCH (부분 수정) / PUT (전체 덮어쓰기)
- DELETE

---
**REST(Representational State Transfer)는 웹에서 자원을 정의하고 접근하는 아키텍처 스타일로, 클라이언트와 서버가 HTTP 프로토콜을 통해 자원을 주고받는 방식입니다.** 즉, URL로 자원을 식별하고, HTTP 메서드(GET, POST, PUT, DELETE 등)로 해당 자원에 대한 행위를 표현하는 구조예요.

📌 **REST의 핵심 정의**

- **아키텍처 스타일**: 특정 기술이나 라이브러리가 아니라, 웹 서비스 설계 원칙을 정의한 개념
- **자원(Resource)**: 웹에서 접근 가능한 모든 데이터(문서, 이미지, 사용자 정보 등)를 자원으로 간주
- **표현(Representation)**: 자원은 JSON, XML, HTML 등 다양한 형태로 표현 가능
- **상태 전이(State Transfer)**: 클라이언트가 서버에 요청을 보내고, 서버가 응답을 반환하면서 애플리케이션 상태가 전이됨

⚙️ **REST의 주요 원칙**

1. **클라이언트-서버 구조**
	- 클라이언트는 사용자 인터페이스 담당, 서버는 데이터 처리 담당
2. **무상태성(Stateless)**
	- 각 요청은 독립적이며, 서버는 이전 요청의 상태를 기억하지 않음
3. **캐시 가능(Cacheable)**
	- 응답은 캐시 가능해야 하며, 성능 향상에 기여
4. **계층 구조(Layered System)**
	- 클라이언트는 중간 서버(프록시, 게이트웨이 등)를 통해 서버와 통신 가능
5. **균일한 인터페이스(Uniform Interface)**
	- 자원은 고유한 URI로 식별되고, HTTP 메서드로 행위를 정의
