
동시 접속자 수와, 사용자(접속자)가 요청하는 초 당 건수로 트래픽을 계산한다.
그 중에서 실제 서버가 처리 해낸 요청 수는 트랜잭션 수로 표현하는 데 이것이 TPS다.

**TPS**(Transactioon per seconds) : 초 당 트랜잭션 수

**트랜잭션**: DB의 ACID 원칙을 지키는 CRUD 행위의 한 작업 단위 크기
=> 데이터 정합성, 일관성에 오류를 일으키지 않도록 완전히 실행된 읽기, 수정, 추가, 삭제 등이 포함된 하나의 작업 단위

## TPS 측정 방법 (운영 단계)

실제 서비스가 운영 중일 때 TPS를 측정하려면 **모니터링 도구**나 **로그 분석**을 활용합니다.

- **애플리케이션 로그 분석**
	- 웹 서버(Nginx, Apache)나 WAS(Tomcat, Spring 등)의 로그에서 초당 요청 수를 집계
	- 예: `access.log`에서 타임스탬프별 요청 건수를 카운트
- **APM(Application Performance Monitoring) 도구 사용**
	- New Relic, Datadog, Prometheus + Grafana 같은 모니터링 툴에서 TPS 지표 제공
	- 초당 처리된 요청/트랜잭션 수를 실시간 그래프로 확인 가능
- **로드 테스트 도구 활용**
	- JMeter, Locust, k6 같은 부하 테스트 툴로 특정 시나리오를 실행하고 TPS 측정
	- 실제 운영 전에도 TPS를 확인할 수 있는 방법

📐 TPS 예상 방법 (기획 단계)

아직 시스템을 만들기 전이라면, **예상 트래픽과 사용자 행동 패턴**을 기반으로 TPS를 추정합니다.

1. **사용자 수 추정**
	- 하루 예상 방문자 수, 동시 접속자 수를 추정
	- 예: 하루 10만 명 방문, 피크 시간 동시 접속자 5000명

2. **행동 패턴 분석**
	- 평균적으로 사용자가 몇 초마다 요청을 발생시키는지 계산
	- 예: 사용자가 페이지 이동/검색/결제 등 평균 10초마다 요청 1건 발생

3. **계산식 적용**
	![[Pasted image 20260204204948.png]]
	- 예: 동시 사용자 5000명, 평균 10초마다 요청 → TPS = 5000 ÷ 10 = 500

4. **여유치 반영**
	- 피크 트래픽(예: 이벤트, 세일 시즌) 고려해 2~3배 버퍼를 잡음
	- 예: 예상 TPS 500 → 설계 시 1500 TPS까지 처리 가능하도록 준비
