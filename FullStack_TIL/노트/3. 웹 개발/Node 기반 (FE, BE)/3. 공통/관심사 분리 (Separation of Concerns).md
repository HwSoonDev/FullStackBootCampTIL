## 웹 개발에서의 관심사 분리 (Separation of Concerns)

### 1.1 관심사 분리란?

**관심사 분리**란 소프트웨어를 구성하는 각 요소가 **하나의 책임(역할)에만 집중하도록 분리하는 설계 원칙**이다.

> 변경 이유가 다른 코드들은 반드시 분리되어야 한다.

---

### 1.2 관심사 분리가 필요한 이유

관심사가 섞일 경우 다음과 같은 문제가 발생한다.

- 코드 가독성 저하
- 수정 시 사이드 이펙트 발생
- 테스트 및 디버깅 어려움
- 협업 시 충돌 증가

관심사 분리를 통해 얻는 효과는 다음과 같다.

- 유지보수성 향상
- 확장성 증가
- 코드 재사용성 개선
- 변경에 강한 구조

---

### 1.3 프론트엔드에서의 관심사 분리

프론트엔드에서는 주로 다음과 같은 기준으로 관심사를 분리한다.

- **데이터 로직**: API 요청, 비동기 처리
- **상태 관리 로직**: 상태 생성 및 변경
- **UI 컴포넌트**: 화면 렌더링
- **스타일**: CSS, 디자인 요소

React에서는 이를 다음과 같이 분리하는 것이 일반적이다.

- service / api layer
- custom hook
- presentational component
- style file

---

### 1.4 백엔드에서의 관심사 분리

백엔드에서는 계층 구조로 관심사를 분리한다.

- **Controller**: HTTP 요청/응답 처리
- **Service**: 비즈니스 로직
- **Repository**: 데이터베이스 접근

이를 통해 각 계층은 명확한 역할만 수행하며, 특정 변경이 전체 시스템에 영향을 주지 않도록 한다.

---

### 1.5 면접 및 이력서 활용 문장 예시

- "관심사 분리를 적용하여 UI, 비즈니스 로직, 데이터 접근 계층을 분리함으로써 유지보수성과 확장성을 개선함"
