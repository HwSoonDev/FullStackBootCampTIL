
세션-쿠키 방식과 별개로 토큰 방식도 두 가지로 나뉜다. 

JWT 기반 인증에서 토큰을 어디에 저장하느냐에 따라  
보안 특성과 라우트 가드 구현 방식이 크게 달라진다.  
XSS, CSRF, 라우트 가드 관점**에서 두 방식을 비교한다.

## 토큰 저장 방식 차이

| 구분 | 로컬 토큰 | 쿠키 토큰 |
|------|-----------|-----------|
| 저장 위치 | localStorage / 메모리 | Cookie (주로 HttpOnly) |
| 요청 시 전송 | JS가 직접 헤더에 추가 | 브라우저가 자동 전송 |
| 서버 미들웨어 접근 | 불가 | 가능 |
결론 요약

|항목|로컬 토큰|쿠키 토큰|
|---|---|---|
|XSS 방어|약함|강함|
|CSRF 방어|강함|추가 대응 필요|
|미들웨어 사용|불가|가능|
|라우트 가드|클라이언트|서버 + 클라이언트|
|UX 안정성|낮음|높음|

## 1) 보안 차이

| 보안 관점 | 🔴로컬 토큰 (LocalStorage / Memory)                                                                | 🟢 쿠키 토큰 (HttpOnly Cookie)                                                                                            |
| ----- | ---------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| XSS   | ❌ 취약 <br><br>토큰이 JS 영역(LocalStorage, 전역 State)에 저장되어 있어 XSS 발생 시 악성 스크립트가 토큰에 직접 접근·탈취할 수 있음   | ✅ 안전<br><br>HttpOnly 옵션으로 인해 JS에서 토큰 접근이 불가능하여 XSS가 발생하더라도 토큰 탈취가 어려움                                                 |
| CSRF  | ✅ 안전<br><br>요청 시 토큰을 JS가 명시적으로 헤더에 실어야 하므로, 외부 사이트에서 발생한 요청에는 인증 정보가 포함되지 않아 CSRF 공격이 성립하기 어려움 | ❌ 취약 <br><br>쿠키는 동일 도메인 요청 시 자동으로 전송되므로, 사용자가 로그인된 상태에서 악성 요청이 발생하면 인증 정보가 함께 전송<br><br>-> samesite 설정을 통해서 대비가 필요하다. |
로컬 토큰은 access token을 로컬, refresh token을 쿠키로 나눠 저장해서 토큰 탈취 위험이 줄어들고, CSRF 공격에 안전하지만, JS 영역에서 다뤄져서 XSS 공격이 취약하다.

반면, 쿠키에 access, refresh 둘다 저장하게 되면, JS영역이 아니기 때문에 XSS 공격에는 안전하고, CSRF 공격은 적절한 대비가 필요하다.

## 2) 라우트 가드(Route Guard) 관점

### 🔴 로컬 토큰 방식

- 서버 미들웨어에서 토큰 확인 불가
- 클라이언트 단 라우트 가드에 의존
- 
```
React Provider - useEffect 기반 인증 체크 - 페이지 진입 후 리다이렉트`
```

❌ **초기 페이지 보호 불가**  
❌ 깜빡임(Flicker) 발생 가능

```js

	const token = req.cookies.get("accessToken") 
}
```

## 6. 실무에서 많이 사용하는 패턴

```
Access Token  → HttpOnly Cookie (짧은 만료 시간) 
Refresh Token → HttpOnly Cookie CSRF Token   → Custom Header
```


- XSS 방어 가능
- CSRF 대응 가능
- Next.js Middleware 기반 라우트 가드 구현 가능

---

## 7. 결론

- 로컬 토큰 방식은 구현이 단순하지만 XSS에 취약하다
- 쿠키 토큰 방식은 CSRF 대응이 필요하지만 서버 라우트 가드에 적합하다
- Next.js 환경에서는 쿠키 기반 JWT + 미들웨어 방식이 실무에서 선호된다
