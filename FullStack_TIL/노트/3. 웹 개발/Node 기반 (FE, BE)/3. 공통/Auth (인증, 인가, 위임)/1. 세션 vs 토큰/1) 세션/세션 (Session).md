## `Session`이란 무엇인가

HTTP는 기본적으로 **요청마다 연결이 끊기는(stateless) 프로토콜**이라, 서버만 보면 “이 요청이 방금 로그인한 사람인지” 알 수 없습니다.  
**세션**은 그 위에 **“이 클라이언트 = 이 서버 쪽 상태”**를 잇기 위한 방법입니다.

핵심은 두 가지입니다.

1. **서버(또는 서버가 쓰는 저장소)에 “세션 데이터”를 둔다.**  
   예: `{ userId: "..." }`, 장바구니, 권한 플래그 등.
2. **클라이언트에는 그 데이터 전체가 아니라 “세션을 가리키는 짧은 식별자”만 준다.**  
   보통 **쿠키**에 `sessionId` 같은 값을 실어 보냅니다.

다음 요청부터 클라이언트가 그 쿠키를내면, 서버는 ID로 저장소에서 세션을 찾아 **같은 사용자로 이어진 요청**처럼 처리합니다.

## 쿠키와의 관계

세션을 꼭 쿠키로만 붙이는 건 아니지만, 브라우저에서는 **쿠키가 가장 흔합니다.**

- 서버 → `Set-Cookie`로 세션 ID 전달  
- 브라우저 → 같은 사이트(또는 설정한 Domain/Path)로 가는 요청에 자동으로 `Cookie` 헤더 첨부  
- **`httpOnly`** 를 켜면 JS에서 읽기 어렵게 해서 XSS로 ID 훔치기 난이도를 올립니다.  
- **`Secure`** 는 HTTPS에서만 전송.  
- **`SameSite`** 는 다른 사이트에서 보내는 요청에 쿠키를 실을지 말지( CSRF·크로스 오리진 API와 연관).

크로스 오리진(프론트 도메인 ≠ API 도메인)이면 **CORS에서 `credentials: true`** 와 프론트의 **`credentials: 'include'`** 가 맞아야 쿠키가 붙습니다.

## express-session에서의 “원리” 한 장 요약

1. 미들웨어가 요청마다 **쿠키의 세션 ID**를 읽는다.  
2. **스토어**(메모리/DB/Redis…)에서 해당 ID의 **직렬화된 세션 객체**를 불러온다.  
3. **`req.session`** 에 올려서 핸들러에서 `req.session.userId = ...` 처럼 쓴다.  
4. 응답을 보낼 때 변경이 있으면 스토어에 다시 저장하고, 필요하면 쿠키를 갱신한다.  
5. **`secret`** 으로 쿠키에 실리는 서명 쿠키 형태를 만들면, 클라이언트가 ID를 조금 바꿔도 서버가 거부할 수 있다(위조 완화).

`saveUninitialized: false` 같은 옵션은 **“아직 아무것도 안 넣은 빈 세션은 DB에 만들지 않는다”**처럼, 저장 비용·쓰레기 세션을 줄이기 위한 동작입니다.

---

## 정리

- **개념:** “서버가 들고 있는 연속된 상태”를 **세션 ID**로 클라이언트와 연결하는 것.  
- **원리:** ID는 쿠키(등)로 왕복하고, **실제 내용은 서버(또는 공유 저장소)** 에 둔다.  
- **이 프로젝트:** ID는 `sid` 쿠키, 내용은 PostgreSQL `session` 테이블, 앱 데이터로는 주로 `userId`를 넣는 **전형적인 서버 사이드 세션**입니다.
