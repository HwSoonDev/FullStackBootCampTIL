
본인이 생각하는 좋은 테스트란 무엇인지 설명해주세요.

### 내 답변

(모범 답안 필사)
좋은 테스트란,
명확하고, 유지 보수가 쉬우며, 소프트웨어 기대 동작을 정확하게 검증해야 합니다.

1) **한번에 하나만 검증**해야합니다. (테스트 목적과 범위 명확성)
   함수 내에 여러 동작이 있다면, 각각 별도의 테스트 케이스로 나누어 검증해야 하며, 이는 테스트의 목적과 범위를 명확하게 해줍니다.
2) **테스트의 이름을 잘 지어야** 합니다. (테스트 명명 규칙)
   테스트의 이름은 해당 테스트가 무엇을 검증하는 지 알 수 있게 해야합니다.
   'testCartShouldBeEmptyAfterInitialization'처럼 구체적인 이름이 좋습니다.
3) 테스트는 **구체적이고, 명시적인 입력**을 사용해야 합니다.
   입력 값을 생성할 때는 단순하고 명확한 값을 사용해야 합니다.
   실제 사례를 반영하고 복잡한 로직은 피해야 합니다.
4) 테스트는 **구현의 세부 사항이 아닌 동작을 검증**해야 합니다. 
   구현의 세세한 차이에 의존하지 않고 시스템의 예상되는 (요구하는) 동작을 해내는 지만 검사해야 합니다. 구현에 대해서는 유연하게 대응하여 코드의 유지보수성을 높입니다.
5) Mock 데이터를 적절하게 사용해야 합니다.
   Mock은 테스트의 독립성을 보장하지만, 과도하게 사용될 경우 실제 기능과 동떨어진 테스트가 될 수도 있습니다. Mock은 최소한으로 사용하고, 실제 코드를 가져올 수 있는지도 고려해야 합니다. Fake으로 구현할 수 있는 부분은 없는지도 고려하여 실제 상황을 모방하는 것이 중요합니다.


### 채점기준

- **테스트의 목적과 범위 명확성** (+2점)
	- 테스트가 한 가지 특정 동작만 검증하는지 설명
- **테스트 명명 규칙** (+2점)
	- 테스트 이름이 테스트 의도를 명확하게 반영하는지 설명
- **구체적이고 명시적인 입력 사용** (+2점)
	- 테스트 입력이 명확하고 구체적으로 제공되는지
	- 입력 생성의 복잡성을 피하고 단순화된 접근 방식 사용 여부
- **테스트 유지 관리성** (+2점)
	- 테스트가 구현 디테일보다 동작을 검증하는지
	- 구현 디테일 변경에 대한 테스트의 유연성
- **Mock의 적절한 사용** (+2점)
	- Mock의 남용을 피하고, 필요한 경우 적절하게 사용하는지
	- Fake 구현의 적용을 고려하여 실제 상황을 얼마나 잘 모방하는지


### (참고) 모범 답안

좋은 테스트는 명확하고, 유지 보수가 쉬우며, 소프트웨어의 기대 동작을 정확하게 검증해야 합니다. 첫째, **한 번에 하나의 동작만 테스트**하는 것이 중요합니다. 이는 테스트의 목적을 명확하게 하고, 실패 시 문제를 빠르게 식별할 수 있게 해줍니다. 예를 들어, 하나의 함수 내 여러 동작이 있더라도 각각을 별도의 테스트 케이스로 분리해야 합니다.

둘째, **테스트의 이름을 잘 지어야** 합니다. 테스트 이름은 해당 테스트가 무엇을 검증하려는지 명확히 해야 하며, 가능한 한 구체적이어야 합니다. 예를 들어, 'testEmptyCart'는 'testCartShouldBeEmptyAfterInitialization'로 더 구체적으로 명명할 수 있습니다.

셋째, 테스트는 **구체적이고 명시적인 입력을 사용**해야 합니다. 입력 값은 테스트의 가독성을 높이고, 코드와 테스트 사이의 관계를 명확하게 해야 합니다. 테스트 입력은 실제 사용 사례를 반영해야 하며, 복잡한 로직을 피해야 합니다.

넷째, 좋은 테스트는 **구현의 세부 사항이 아닌 동작을 검증**해야 합니다. 테스트는 가능한 한 구현에 묶이지 않고, 시스템의 예상되는 동작을 검증해야 합니다. 이는 구현 방법이 변경되어도 테스트를 재작성할 필요가 없게 합니다.

마지막으로, **Mock의 사용은 신중하게** 해야 합니다. Mock은 테스트의 독립성을 보장하지만, 과도하게 사용되면 실제 코드의 동작을 잘못 반영할 수 있습니다. 필요한 경우, 실제 구현을 사용하거나, 가능한 최소한으로 Mock을 사용하고, 가능하다면 Fake 구현을 고려해야 합니다. 이는 테스트를 현실적인 상황에 더 가깝게 만들어줍니다.
